Migration analysis

Move this repo from Fable/F# to plain modern web tech.

The codebase is already very close to a no-build architecture. It serves static HTML pages, uses direct DOM manipulation, stores data in localStorage, and has almost no framework-level state management. The build step exists mostly to translate F# and bundle it, not because the browser needs it.

Bottom line

Yes, migrate.

For this specific project, plain ES modules plus native HTML features is the simplest long-term fit.

Recommended target

Static HTML + ESM

No bundler in the default workflow. Optional tests only. Optional type checking only through JSDoc or a one-off tsc --checkJs.

Use arrow keys, space, or the buttons. 1 / 10

  • src/App.fs builds shared nav HTML and dispatches to page-specific init functions.

Important observation

The app is not using F# for a big architectural advantage such as a message loop, server/shared contracts, or complex domain modeling. It is mostly using F# as a nicer authoring language for browser DOM code.

That makes migration much easier.

Most of the runtime complexity is tooling, not app behavior. 2 / 10

Behavioral inventory

What the app actually does

Text utilities

  • Compare.fs: split, trim, distinct, set math, CSV download.
  • Unique.fs: duplicate detection.
  • Babynames.fs: local preference tracking with a static array dataset.

Small calculators

  • Days.fs: date math and persistence.
  • Alcohol.fs: arithmetic, list rendering, local persistence.
  • Equation.fs: random expression generation and answer checking.

Micro interactions

  • Memory.fs: temporary game state and timers.
  • Heartbeat.fs: keyboard/button input and timing.
  • Dialog.fs: modal UI currently generated with Feliz.ViewEngine.

Translation implication

None of these features need a framework. They map directly to browser primitives: querySelector, addEventListener, <dialog>, localStorage, crypto.randomUUID(), navigator.clipboard.writeText(), URL, Intl, and plain arrays/objects.

The project is a bundle of independent widgets, not a single SPA. 3 / 10

Why migrate

What is costing you today

Cost centers

  • Two toolchains to keep alive: .NET and npm.
  • Generated artifacts obscure what really ships.
  • Debugging flows through F# source, generated JS, and webpack output.
  • Simple DOM work becomes coupled to Fable APIs and emitted helpers.
  • bundle.js ships the whole app to every page.

Mismatch with your preference

  • You do not want a type-heavy authoring model for this kind of project.
  • You are already relying on behavior tests and manual testing rather than type-level guarantees.
  • The repo's problem domain is mostly UI plumbing and small transformations, where plain JavaScript stays readable.

Important nuance

The migration is worth it because the current app is small and page-oriented. If this were a large shared-domain system with APIs, cross-platform logic, or hard business invariants, the answer would be less obvious.

This is a good candidate for simplification, not just a rewrite for novelty. 4 / 10

Recommendation

Target architecture

Core approach

  • Keep static per-page HTML files in public/.
  • Replace the global bundle with per-page <script type="module">.
  • Use a tiny shared web/ or public/js/ folder for helpers.
  • Use native browser APIs directly, not a rendering library.
  • Use native <dialog> instead of the custom modal renderer.

Modern HTML features

Native replacements for custom code

Replace this

  • Feliz.ViewEngine-generated nav and modal markup.
  • Manual modal backdrop visibility classes.
  • Some custom UI composition code in App.fs and Dialog.fs.

With this

  • <template> for reusable page shell fragments if needed.
  • <dialog> plus showModal() for confirms and info dialogs.
  • <script type="module"> for native imports.
  • input[type=date], input[type=number], dataset, form events, and crypto.randomUUID().
// example shape, not final code
import { mountShell } from "./app-shell.js";
import { confirmDialog } from "./dialog.js";

document.querySelector("#clear-drinks").addEventListener("click", async () => {
  const ok = await confirmDialog({
    message: "Clear all tracked drinks?",
    phrase: "delete all"
  });
  if (!ok) return;
  localStorage.removeItem("alcohol-drinks");
});
Modern HTML can remove real code here, not just shift it around. 7 / 10

Migration plan

Smallest safe path

Phase 1

  • Create shared JS helpers for DOM lookup, storage, clipboard, and number steppers.
  • Move nav generation from App.fs to app-shell.js.
  • Replace Dialog.fs with a native <dialog>-based helper.

Phase 2

  • Migrate one easy page first: unique or compare.
  • Load only that page's module from its HTML file.
  • Keep Fable running for the untouched pages during the transition.

Phase 3

  • Migrate the remaining pages one by one.
  • Move the static names dataset from Names.fs into a plain JS module or JSON file.
  • Delete F# files only after their JS replacement is active.

Phase 4

  • Remove dotnet, Fable, webpack, generated JS, and the giant shared bundle.
  • Simplify deploy to straight file sync of public/.
  • Add optional browser tests if you want regression safety.
Migrate page-by-page, not rewrite-all-at-once. 8 / 10

Risks

What to watch out for

Main risks

  • Behavior drift: page logic is simple but user-visible, so small regressions matter.
  • Shared helpers: do not recreate a mini framework while trying to escape one.
  • Data compatibility: preserve existing localStorage keys like theme, liked, disliked, and alcohol-drinks.
  • Module paths: no-build means being explicit and consistent with import paths.

Risk reducers

  • Keep HTML ids and storage keys stable during migration.
  • Migrate one page at a time and compare behavior side by side.
  • Add a very small Playwright smoke suite for critical flows.
  • Use JSDoc only where it improves editor feedback, not everywhere.

Optional testing stance that matches your preference

Use tests as the main safety net. Keep typing optional and lightweight. A practical setup would be browser smoke tests for each page plus a few direct unit tests for pure helpers like list comparison and date math.

The trap is overengineering the replacement. 9 / 10

Decision

Recommended migration statement

Recommended

Rebuild the repo as static HTML plus per-page ES modules, using native browser APIs and modern HTML features, with no bundler in the normal development path.

Why this is the right simplification

  • The current app is page-oriented and mostly imperative DOM logic already.
  • The biggest complexity comes from the authoring toolchain, not the app itself.
  • This keeps deployment static and cheap.
  • This matches your preference for less ceremony and less type-driven development.

First migration candidate

Start with unique or compare. They are pure, small, and reveal the shared helper needs quickly without pulling in the modal, timers, or dataset migration first.
